## Management Summary

### Verdict

🔴 Not production-ready. Juice Shop is a training target - intentional weaknesses live in every tier. Anyone with a browser and public repo read access can take over admin accounts and execute code on the server through several independent paths, none of which require elevated privilege or special tooling. Fixing this means rebuilding authentication primitives, data-access query construction, and secret management - not patching a single bug.

**Risk distribution:** 🔴 Critical: 10 · 🟠 High: 17 · 🟡 Medium: 5 · 🟢 Low: 2 · **Total: 34**

<br/>

**Worst-case scenarios behind this verdict - what an attacker can do today:**

<blockquote style="border-left: 3px solid #dc2626; background: #fef2f2; padding: 16px 20px; margin: 0;">

- **Forge admin credentials without knowing any password** — The RSA private key used to sign session tokens is committed to the public repository, so anyone who reads the source can mint an admin token and access every privileged API endpoint. *(🔴 [F-001](#f-001) — JWT forgery via hardcoded RSA private key lib/insecurity.ts, 🔴 [F-009](#f-009) — JWT algorithm none bypass via express jwt 0.1.3 lib/insecurity.ts)*
- **Log in as admin via a one-line SQL payload** — The login form's query interpolates user input directly into a raw SQL string; a single injected clause bypasses password verification and returns the seeded admin account. *(🔴 [F-002](#f-002) — SQL injection login bypass routes/login.ts, 🔴 [F-010](#f-010) — User privilege escalation via SQL injection routes/login.ts)*
- **Execute code on the server from a product order** — The B2B order endpoint evaluates attacker-supplied strings through a bypassable JavaScript sandbox, and the user-profile endpoint passes the username field through a bare eval call — both reachable with a forged token. *(🔴 [F-007](#f-007) — SSTI via eval in username field routes/userProfile.ts, 🔴 [F-008](#f-008) — RCE via safeEval bypass in B2B order routes/b2bOrder.ts)*
- **Read internal files via an uploaded XML document** — The file-upload endpoint parses XML with external-entity resolution enabled, allowing an uploaded document to read server-side files such as credentials or configuration. *(🔴 [F-004](#f-004) — XXE via libxmljs2 XML file upload routes/fileUpload.ts)*
- **Hijack any user session via stored cross-site scripting** — Six frontend components render user-controlled HTML without sanitization; injected scripts run in the full session context and can steal the stored login token for any visitor. *(🔴 [F-012](#f-012) — Stored XSS via trust HTML bypass in about component about.component.ts, 🔴 [F-013](#f-013) — XSS via last login IP header injection last login ip.component.ts, 🔴 [F-018](#f-018) — JWT token storage in localStorage app.guard.ts)*

</blockquote>

<br/>

Replacing the committed signing key, parameterizing the login and search queries, removing both eval paths, and upgrading password hashing to bcrypt are the minimum prerequisites for any non-training deployment.

**Attack-chain analysis.** 5 findings anchor 4 code-verified attack chains (see [§9](#9-abuse-cases) Abuse Cases): 🔴 [F-001](#f-001) — JWT forgery via hardcoded RSA private key, 🔴 [F-007](#f-007) — SSTI via eval in username field, 🔴 [F-010](#f-010) — User privilege escalation via SQL injection, 🔴 [F-012](#f-012) — Stored XSS via trust HTML bypass in about component about.component.ts, 🔴 [F-018](#f-018) — JWT token storage in localStorage app.guard.ts. 2 of these are rated above their individual baseline as a direct result of the verified chain.

### 3.1 JWT forgery via hardcoded RSA private key lib/insecurity.ts

**Source:** 🔴 [F-001](#f-001) — `lib/insecurity.ts:23`

Severity **Critical** ([CWE-321](https://cwe.mitre.org/data/definitions/321.html)). STRIDE: Spoofing. See [§8 T-001](#t-001) for the full register row.

**Attack Steps**

1. The RSA private key is hardcoded at `lib/insecurity.ts:23` and committed to the public GitHub repository.
2. Any attacker with access to the source code can use `jwt.sign({ data: { id: 1, email: 'admin@juice-sh.op', role: 'admin' } }, privateKey, { algorithm: 'RS256' })` to create a valid administrative JWT.
3. This gives full admin access to all API endpoints requiring authentication.

**How to read the verdicts.** Every control category (and every sub-control below it) carries exactly one status. The two red verdicts do **not** mean the same thing - this is the distinction that decides what you have to do about a finding:

| Status | Meaning | What it asks of you |
|----------|------------------------------------|------------------------|
| 🟢 Adequate | Control is present and sound | Nothing - keep it |
| 🟡 Partial | Present, but with meaningful gaps | Close the gap |
| 🟠 Weak | Present, but has exploitable gaps | Strengthen it |
| 🔴 Unsafe | **Present and relied upon, but defeated / trivially bypassable** | **Fix the existing control** |
| 🔴 Missing | **Control was never built** | **Add the control** |
| - | Not applicable to this codebase | - |

So "🔴 Unsafe" on a control category does *not* mean the control is absent - it means the control exists but does not hold (e.g. an `MD5` password hash, a raw-SQL query path, a hardcoded signing key). "🔴 Missing" is reserved for controls that were never built (e.g. no Content-Security-Policy header).

<a id="password-based-authentication"></a>
#### 7.2.1 Password-Based Authentication

**Status:** 🟠 Weak - SQL injection in the login query allows authentication bypass without knowing any password; `MD5` hashing makes stolen credentials recoverable by GPU dictionary attack within seconds offline.

`routes/login.ts` handles credential verification by querying the SQLite Users table. On a successful match, `lib/insecurity.ts:issueAuthToken()` signs a `{ id, email, role }` payload with the RSA private key and returns an `RS256` JWT. Password reset flows through `routes/resetPassword.ts` and `routes/changePassword.ts`; the TOTP second factor is verified in `routes/2fa/verify.ts` using `otplib`.

The diagram shows the positive password-login path, including the optional TOTP branch:

```mermaid
sequenceDiagram
    autonumber
    actor User
    participant API as Login Route
    participant DB as SQLite Users
    participant TOTP as 2FA Verify Route
    participant JWT as issueAuthToken()

    User->>API: POST /rest/user/login {email, password}
    API->>DB: Query Users WHERE email='...' AND password=md5('...')
    DB-->>API: User row (if match)
    alt TOTP enrolled
        API-->>User: { tmpToken, usingTotp: true }
        User->>TOTP: POST /rest/2fa/verify {tmpToken, totp}
        TOTP->>JWT: Sign payload {id, email, role}
        JWT-->>TOTP: RS256 JWT
        TOTP-->>User: { token, bid, email }
    else No TOTP
        API->>JWT: Sign payload {id, email, role}
        JWT-->>API: RS256 JWT
        API-->>User: { token, bid, email }
    end
```

**Security assessment**

The login query is built as a raw SQL string at `routes/login.ts:34`:

```ts
models.sequelize.query(`SELECT * FROM Users WHERE email = '${req.body.email}' AND password = '${req.body.password}' AND deletedAt IS NULL`)
```

- **SQL injection** — the `email` field is interpolated without escaping; the payload `' OR '1'='1'--` returns the first user row (the seeded admin account) regardless of the password supplied. The same route also exposes the full database schema via the `sqlite_master` table when `SELECT *` returns without a WHERE match (🟠 [F-020](#f-020) — Database schema disclosure via SQLite master table routes/search.ts).
- **Password hashing** — `lib/insecurity.ts` hashes with `crypto.createHash('md5').update(password).digest('hex')` — unsalted `MD5`. A leaked Users table yields crackable hashes in seconds with any GPU dictionary attack. Deep dive at [§7.9.2 Password Storage](#password-storage).
- **CSRF on password change** — `routes/changePassword.ts` accepts the new password via GET parameter with no CSRF token (🟠 [F-027](#f-027) — CSRF on password change routes/changePassword.ts).
- **Rate limiting** — `express-rate-limit` 7.5.1 is applied to selected endpoints but not to `/rest/user/login` or `/rest/resetPassword`. Brute-force attacks against credentials are unrestricted on the primary authentication entry point; given the `MD5` hash weakness, an attacker can combine online credential stuffing with offline hash cracking without friction (🟠 [F-025](#f-025) — No rate limiting on login endpoint server.ts).

**Relevant findings**

- 🔴 [F-002](#f-002) — SQL injection in the login query enables authentication bypass without knowing any password.
- 🟠 [F-020](#f-020) — Database schema disclosed via the `sqlite_master` table in the search route.
- 🟠 [F-025](#f-025) — No rate limit on the login endpoint combined with MD5 hashing enables unrestricted online brute-force and offline dictionary attacks.

#### 7.2.2 Multi-Factor Authentication

**Status:** 🟡 Partial - TOTP enrollment and verification are implemented correctly via `otplib`, but MFA is optional and not enforced on the admin account.

TOTP-based second factors protect accounts that have chosen to enroll. `routes/2fa/setup.ts` generates a TOTP secret and QR code using `otplib`; `routes/2fa/verify.ts` validates the one-time code against the stored secret. Once enrolled, the login flow issues a short-lived `tmpToken` instead of the full session JWT and requires the TOTP code before completing authentication.

The diagram shows the TOTP enrollment and verification flow:

```mermaid
sequenceDiagram
    autonumber
    actor User
    participant Setup as 2FA Setup Route
    participant Verify as 2FA Verify Route
    participant DB as SQLite Users
    participant JWT as issueAuthToken()

    User->>Setup: POST /rest/2fa/setup (authenticated)
    Setup-->>User: TOTP secret + QR code URI
    User->>Setup: POST /rest/2fa/setup {token: TOTP code}
    Setup->>DB: Store totpSecret for user
    DB-->>Setup: OK
    Setup-->>User: 2FA enabled

    Note over User,JWT: On next login
    User->>Verify: POST /rest/2fa/verify {tmpToken, totp}
    Verify->>DB: Load totpSecret, verify TOTP code
    Verify->>JWT: Sign {id, email, role}
    JWT-->>Verify: RS256 JWT
    Verify-->>User: { token, bid, email }
```

**Security assessment**

`otplib` TOTP verification is correctly implemented - the code is time-windowed and verified against the stored secret. The structural gap is that enrollment is voluntary and the admin account in the seeded database has no TOTP secret set, meaning any admin JWT forgery (via 🔴 [F-001](#f-001) — JWT forgery via hardcoded RSA private key lib/insecurity.ts or 🔴 [F-009](#f-009) — JWT algorithm none bypass via express jwt 0.1.3 lib/insecurity.ts) bypasses this control entirely. the `tmpToken` issued mid-flow is itself a JWT signed with the same hardcoded key, so an attacker who has already forged a full admin token never enters the TOTP branch at all.

**Relevant findings**

- 🟠 [F-011](#f-011) — Client-side JWT forgery bypasses the TOTP branch because attackers issue a full admin token directly, never receiving a `tmpToken`.

**Verdict:** 🟠 Weak

<!-- The line below is mechanically derived from the controls table — LLM must not re-author it. -->
**Controls covered:** [7.3.1 Session Token Issuance](#731-session-token-issuance)

- [7.3.1 Session Token Issuance](#session-token-issuance)

**Implemented controls:** `RS256`-signed JWTs issued on every successful login; `express-jwt` 0.1.3 middleware validates tokens on protected routes; token expiry set to one hour.

**Assessment:** This application uses a single locally-signed token format (commonly called JWT) for every authenticated session, regardless of the login flow in [§7.2](#72-identity-and-authentication-controls) that established it. The sub-sections below trace one token through its lifecycle: signing on issuance, validation on every protected request, storage in the browser, manual revocation, and time-based expiry. The signing key is committed in plaintext at `lib/insecurity.ts:23`; the verifier library accepts `alg:none`; and tokens are stored in `localStorage` rather than an HttpOnly cookie - all three weaknesses interact to make the session control boundary entirely bypassable.

<a id="session-token-issuance"></a><a id="session-token-issuance-jwt-based"></a>
#### 7.3.1 Session Token Issuance

**Status:** 🟠 Weak - the signing key is committed to the public repository and the verifier library accepts `alg:none`, making signature verification moot.

`lib/insecurity.ts:issueAuthToken()` signs a `{ id, email, role }` payload with an RSA-2048 private key using the `RS256` algorithm, setting a one-hour expiry. Both `routes/login.ts` and `routes/2fa/verify.ts` call this single helper, so token format and lifetime are consistent across flows.

The diagram shows the positive issuance path from a verified credential to the encoded JWT returned to the browser:

```mermaid
sequenceDiagram
    autonumber
    actor User
    participant API as Login Route
    participant Issuer
    Note over Issuer: issueAuthToken()
    participant Key
    Note over Key: RSA Private Key (lib/insecurity.ts:23)

    User->>API: Successful credential check
    API->>Issuer: Sign payload {id, email, role}
    Issuer->>Key: Load private key (hardcoded string)
    Key-->>Issuer: PEM bytes
    Issuer-->>API: Encoded JWT (RS256, 1h)
    API-->>User: { token, bid, email }
```

**Security assessment**

The intended control is asymmetric signature on a short-lived token. That boundary is defeated at two independent points:

- **Hardcoded signing key** — `lib/insecurity.ts:23` contains the full RSA private key as a string literal committed to the public repository. Anyone who can read the source can sign arbitrary tokens that the server accepts as genuine.
- **Algorithm confusion** — `express-jwt` 0.1.3 does not pin the algorithm; a token signed with `alg:none` (no signature) passes the verifier, making the committed key redundant as an attack primitive.
- **Key exposure** — the matching public key is served at `/encryptionkeys/jwt.pub` over HTTP, and the full `encryptionkeys/` directory is browsable without authentication (🟠 [F-019](#f-019) — JWT public key and encryption keys publicly browsable server.ts).

The hardcoded private key that breaks the issuance boundary appears at:

```ts
export const privateKey =
  '[PEM PRIVATE KEY — REDACTED]
<REDACTED — key bytes masked>
-----END RSA PRIVATE KEY-----'
```

**Relevant findings**

- 🔴 [F-001](#f-001) — Committed RSA private key allows any attacker to forge RS256 admin tokens.
- 🔴 [F-009](#f-009) — `express-jwt` 0.1.3 accepts `alg:none`, providing a key-independent forgery path.
- 🔴 [F-018](#f-018) — JWT stored in `localStorage` exposes the token to any XSS payload executing in the same origin.
- 🟠 [F-019](#f-019) — The encryption-keys directory is publicly browsable, disclosing the public key and premium content key.

#### 7.5.1 SQL Query Parameterization

**Status:** 🟠 Weak - two critical routes bypass the Sequelize ORM and call raw `models.sequelize.query()` with user-controlled template literals.

Sequelize 6.37 backs the SQLite data layer and uses parameterized queries (`?` placeholders with `replacements`) for all auto-generated model finder queries. The login route and the product-search route are exceptions: both call `models.sequelize.query()` directly.

The vulnerable login lookup is built as a raw SQL string:

```ts
models.sequelize.query(
  `SELECT * FROM Users WHERE email = '${req.body.email}' AND password = '${req.body.password}' AND deletedAt IS NULL`
)
```

**Security assessment**

- **Login SQL injection** (🔴 [F-002](#f-002) — SQL injection login bypass routes/login.ts) — `routes/login.ts:34` interpolates `req.body.email` and `req.body.password` without escaping. The payload `' OR '1'='1'--` bypasses password verification and returns the seeded admin row.
- **Search SQL injection** (🔴 [F-003](#f-003) — SQL injection in product search routes/search.ts) — `routes/search.ts` interpolates `req.query.q` into a `LIKE` clause in the same raw-query pattern; `%' UNION SELECT ...--` enables full data exfiltration and schema enumeration.
- Both routes could be replaced with `Model.findOne({ where: { email, password } })` or `Model.findAll({ where: { name: { [Op.like]: `%\${q}%` } } })`, which use bound parameters by construction.

**Relevant findings**

- 🔴 [F-002](#f-002) — SQL injection in the login query enables authentication bypass and admin account takeover.
- 🔴 [F-003](#f-003) — SQL injection in the product-search query enables data exfiltration and schema disclosure.
- 🟠 [F-020](#f-020) — Database schema disclosed via `sqlite_master` table accessible through the injectable search route.

<a id="nosql-query-construction"></a>
#### 7.5.2 NoSQL Query Construction

**Status:** 🟠 Weak - MarsDB selectors in the order-tracking route accept operator objects (`$where`) where only scalar string IDs should be valid.

MarsDB (in-memory MongoDB-compatible store) is used for order and basket data. `routes/trackOrder.ts` queries orders by `orderId` from `req.params.id`.

**Security assessment**

The MarsDB selector in `routes/trackOrder.ts` does not enforce scalar type on the `orderId` parameter. Passing a JSON object `{ "$where": "1 == 1" }` via the URL satisfies the MarsDB query and returns all order records (🔴 [F-021](#f-021) (NoSQL injection via \$where operator)). A similar pattern on the product-review update path (`routes/reviews.ts`) accepts a user-controlled selector in the MarsDB update call, allowing cross-user review modification (🔴 [F-010](#f-010) — User privilege escalation via SQL injection routes/login.ts).

`about.component.ts` binds the `about` field from the backend to `[innerHTML]` via `bypassSecurityTrustHtml()` (🔴 [F-012](#f-012) — Stored XSS via trust HTML bypass in about component about.component.ts). Any attacker who can write the backend's `about` value (admin API access is achievable via JWT forgery) can inject a persistent XSS payload served to every visitor. `last-login-ip.component.ts` reflects the server's `X-Last-Login-IP` header through the same bypass (🔴 [F-013](#f-013) — XSS via last login IP header injection last login ip.component.ts), making the header value an injection point for stored reflective XSS. Since tokens are stored in `localStorage` (see [§7.3](#73-session-and-token-controls)), any XSS payload that reads `localStorage.getItem('token')` exfiltrates the victim's session JWT.

**Relevant findings**

- 🔴 [F-012](#f-012) — Stored XSS via `bypassSecurityTrustHtml()` in the about component; attacker-controlled HTML rendered to all visitors.
- 🔴 [F-013](#f-013) — Reflected XSS via the `X-Last-Login-IP` header rendered without escaping in `last-login-ip.component.ts`.

<a id="cors-policy"></a>
#### 7.8.1 CORS Policy

**Status:** 🟠 Weak - `app.use(cors())` is registered at `server.ts` with no `origin` option, allowing all origins to make credentialed cross-origin requests.

CORS headers are applied globally by the `cors` npm package, invoked at `server.ts` immediately after Helmet. `app.options('*', cors())` accepts cross-origin preflight for every route.

**Relevant findings**

- 🟠 [F-016](#f-016) — Wildcard CORS allows any origin to make credentialed API requests, enabling cross-origin session data exfiltration.

**Status:** 🔴 Missing - no `Content-Security-Policy` header is present; Helmet 4.6 does not enable CSP by default and `helmet.contentSecurityPolicy()` is not called in `server.ts`.

Helmet 4.6 is imported at `server.ts` and called with `app.use(helmet())`. Other Helmet modules (`noSniff`, `frameguard`, `xssFilter`) activate automatically under the default profile.

**Security assessment**

`Content-Security-Policy` is absent from all responses. Without a CSP, the browser imposes no restriction on which scripts, stylesheets, or image sources the page may load. The stored XSS findings at 🔴 [F-012](#f-012) — Stored XSS via trust HTML bypass in about component about.component.ts and 🔴 [F-013](#f-013) — XSS via last login IP header injection last login ip.component.ts execute injected scripts in the full origin context; a CSP with `script-src 'self'` would prevent inline script execution and significantly reduce the XSS impact. Enabling CSP is the single highest-leverage missing browser control given the XSS surface present.

**Relevant findings**

- 🟠 [F-016](#f-016) — Absence of CSP amplifies the impact of stored-XSS findings by allowing injected scripts to run without restriction.

<a id="csrf-protection"></a>
#### 7.8.3 CSRF Protection

**Status:** 🔴 Missing - no CSRF token is issued or validated on any state-changing endpoint; `routes/changePassword.ts` accepts the new password via a GET request parameter.

**Security assessment**

**Relevant findings**

### 7.9 Cryptography Secrets and Data Protection

**Verdict:** 🟠 Weak

<!-- The line below is mechanically derived from the controls table — LLM must not re-author it. -->
**Controls covered:** [7.9.1 Secret and Key Management](#791-secret-and-key-management) · [7.9.2 Password Storage](#792-password-storage)

- [7.9.1 Secret and Key Management](#secret-and-key-management)

**Implemented controls:** RSA-2048 chosen as the JWT signing algorithm (appropriate key size and algorithm class); distroless runtime container prevents credential extraction via package manager.

**Assessment:** The algorithm choices are sound in principle - RSA-2048 and bcrypt would each be adequate controls. The implemented reality is unsalted `MD5` for passwords and a committed RSA private key for JWT signing. Both weaknesses are structural: the hash algorithm is the wrong choice, and key management was never separated from source code.

<a id="secret-and-key-management"></a><a id="secret-management"></a>
#### 7.9.1 Secret and Key Management

**Status:** 🟠 Weak - the RSA private key used to sign JWTs is committed to the public repository as a string literal; no runtime secret injection or key rotation mechanism exists.

`lib/insecurity.ts` exports `privateKey` and `publicKey` as module-level string constants derived directly from embedded PEM literals. The same file exports `cookieSecret` and several other credential-equivalent values.

**Security assessment**

Two weaknesses converge on the same module:

- **Committed signing key** (🔴 [F-001](#f-001) — JWT forgery via hardcoded RSA private key lib/insecurity.ts) — the RSA private key is a string literal in a public repository. Anyone who reads `lib/insecurity.ts:23` can generate valid admin JWTs without touching the running server.
- **Key browsable at runtime** (🟠 [F-019](#f-019) — JWT public key and encryption keys publicly browsable server.ts) — `server.ts` serves the `encryptionkeys/` directory statically; the public key, premium content key, and associated metadata are downloadable without authentication from `/encryptionkeys/`.

No key rotation mechanism exists - the same key value is baked into the application at build time. Remediation requires replacing the literal with a runtime-injected secret (e.g., via environment variable or a secrets manager) and removing the `encryptionkeys/` static route.

**Relevant findings**

- 🔴 [F-001](#f-001) — RSA private key committed at `lib/insecurity.ts:23`; directly enables JWT forgery.
- 🔴 [F-009](#f-009) — `express-jwt` 0.1.3 accepts `alg:none`; the committed key is not even required to forge a valid token.
- 🟠 [F-019](#f-019) — Encryption-keys directory is publicly browsable, disclosing the JWT public key and premium content key.

<a id="password-storage"></a>
#### 7.9.2 Password Storage

**Status:** 🟠 Weak - user passwords are hashed with unsalted `MD5`; no cost factor is applied; a leaked Users table yields crackable hashes in seconds.

Password registration and login both call `lib/insecurity.ts:hash()`, which wraps `crypto.createHash('md5').update(password).digest('hex')`. The function is used for both the initial hash on registration and the comparison hash on login.

The password hashing call that bypasses any meaningful resistance:

```ts
export const hash = (data: string) => crypto.createHash('md5').update(data).digest('hex')
```

**Security assessment**

`MD5` is a general-purpose message-digest algorithm - it is fast by design and has no cost-factor parameter. A GPU running hashcat processes hundreds of millions of `MD5` hashes per second. The absence of a salt means identical passwords produce identical hashes, enabling rainbow-table attacks across all user accounts simultaneously. The correct control - bcrypt, scrypt, or Argon2id with a suitable cost factor - is a one-line change to the `hash()` helper and would require no other route changes.

**Relevant findings**

- 🔴 [F-006](#f-006) — Unsalted MD5 password hashing; leaked Users table crackable with commodity hardware in seconds.

#### 7.11.2 Runtime Code Execution Controls

**Status:** 🟠 Weak - two request handlers call `eval()` or `safeEval()` on attacker-controlled strings; the distroless image prevents shell execution but not Node\.js-level RCE.

`routes/b2bOrder.ts` passes the `orderLinesData` field from a POST request body through `safeEval()` (the `notevil` sandbox library). `routes/userProfile.ts` passes the username field from the Pug template context through `eval()`. Both are reachable via an authenticated (forgeable) JWT.

**Security assessment**

**Relevant findings**

- 🔴 [F-007](#f-007) — SSTI via `eval()` in username field; direct Node\.js code execution in the user-profile route.
- 🔴 [F-008](#f-008) — RCE via `safeEval` (notevil) bypass in the B2B order route.

### 7.13 Defense-in-Depth Summary

**Verdict:** 🟠 Weak

Two positive controls provide genuine depth in this codebase: the distroless runtime container (`gcr.io/distroless/nodejs24-debian13`) eliminates the shell attack surface that a standard Node image carries, and the RSA-2048 algorithm choice for JWT signing is sound at the algorithm level - the weakness is key management, not the algorithm. All three control boundaries that need structural repair converge on the same theme: trust was placed in source-code constants (signing key), raw string interpolation (query construction), and runtime code evaluation (B2B orders, user profiles) where parameterized values, runtime-injected secrets, and declarative data structures respectively should have been used. Replacing the hardcoded JWT signing key with a runtime-injected value, switching the login and search routes to Sequelize parameterized queries, removing the two `eval()` paths, and adding `bcrypt` password hashing would close the Critical findings and restore meaningful layered defense across the authentication, data-access, and code-execution boundaries simultaneously.

<a id="appendix-a-vektor-taxonomy"></a>
## Appendix A — Vektor Taxonomy

This appendix defines the attacker-starting-position labels used in the Top Threats table and throughout [§8 Findings Register](#8-findings-register). Each label answers the question *what does the attacker need before the exploit begins?*

<a id="vektor-internet-anon"></a>
### Internet Anon

**Attacker position:** Unauthenticated attacker from the public internet · **Breach distance:** 1

**Preconditions:**

- Endpoint is reachable from the internet (no IP allowlist, no VPN)
- No authentication middleware blocks the request

**Typical CWEs:** [CWE-89](https://cwe.mitre.org/data/definitions/89.html) · [CWE-79](https://cwe.mitre.org/data/definitions/79.html) · [CWE-306](https://cwe.mitre.org/data/definitions/306.html) · [CWE-327](https://cwe.mitre.org/data/definitions/327.html) · [CWE-611](https://cwe.mitre.org/data/definitions/611.html) · [CWE-918](https://cwe.mitre.org/data/definitions/918.html)

**Typical OWASP Top 10:** A01:2021, A03:2021, A07:2021

<a id="vektor-internet-user"></a>
### Internet User

**Attacker position:** Any authenticated low-privilege user (valid JWT / session) · **Breach distance:** 2

**Preconditions:**

- Attacker has signed up or otherwise obtained a valid user session
- Endpoint is behind auth but not behind role/admin checks

**Typical CWEs:** [CWE-434](https://cwe.mitre.org/data/definitions/434.html) · [CWE-611](https://cwe.mitre.org/data/definitions/611.html) · [CWE-918](https://cwe.mitre.org/data/definitions/918.html) · [CWE-352](https://cwe.mitre.org/data/definitions/352.html) · [CWE-287](https://cwe.mitre.org/data/definitions/287.html)

**Typical OWASP Top 10:** A01:2021, A04:2021, A05:2021, A10:2021

<a id="vektor-internet-priv-user"></a>
### Internet Priv User

**Attacker position:** Authenticated admin-level user (JWT with admin role or equivalent) · **Breach distance:** 2

**Preconditions:**

- Attacker holds admin credentials or has elevated privileges
- Endpoint gated on admin role but still exploitable once reached

**Typical CWEs:** [CWE-862](https://cwe.mitre.org/data/definitions/862.html) · [CWE-79](https://cwe.mitre.org/data/definitions/79.html) · [CWE-94](https://cwe.mitre.org/data/definitions/94.html)

**Typical OWASP Top 10:** A01:2021

<a id="vektor-victim-required"></a>
### Victim-Required

**Attacker position:** Attacker needs victim interaction - social engineering, crafted link, or live session · **Breach distance:** 2

**Preconditions:**

- Victim must click a link, load a page, or have an active session
- Applies to XSS, CSRF, click-jacking, open redirect

**Typical CWEs:** [CWE-79](https://cwe.mitre.org/data/definitions/79.html) · [CWE-352](https://cwe.mitre.org/data/definitions/352.html) · [CWE-601](https://cwe.mitre.org/data/definitions/601.html) · [CWE-1021](https://cwe.mitre.org/data/definitions/1021.html)

**Typical OWASP Top 10:** A01:2021, A03:2021

<a id="vektor-build-time"></a>
### Build-Time

**Attacker position:** Attacker controls a build input - CI runner, dependency, base image, or external data fetched during build · **Breach distance:** 3

**Preconditions:**

- Compromise of a dependency, registry, or base image
- OR compromise of a CI runner with write access to artifacts

**Typical CWEs:** [CWE-506](https://cwe.mitre.org/data/definitions/506.html) · [CWE-829](https://cwe.mitre.org/data/definitions/829.html) · [CWE-1039](https://cwe.mitre.org/data/definitions/1039.html) · [CWE-1104](https://cwe.mitre.org/data/definitions/1104.html)

**Typical OWASP Top 10:** A08:2021

<a id="vektor-repo-read"></a>
### Repo-Read

**Attacker position:** Attacker gains read access to source repository (leaked clone, forked fork, insider, compromised developer workstation) · **Breach distance:** 3

**Preconditions:**

- Read access to the source tree at or after commit time
- No runtime exploit needed - the vulnerability is the content of the repo

**Typical CWEs:** [CWE-798](https://cwe.mitre.org/data/definitions/798.html) · [CWE-312](https://cwe.mitre.org/data/definitions/312.html) · [CWE-540](https://cwe.mitre.org/data/definitions/540.html)

**Typical OWASP Top 10:** A02:2021, A07:2021

<a id="vektor-n-a"></a>
### n/a

**Attacker position:** Architectural / meta-finding - no runtime entry point, the finding describes a design defect aggregating multiple code-level findings

**Preconditions:**

- Finding is AF-NNN (architectural) rather than F-NNN (code-level)
