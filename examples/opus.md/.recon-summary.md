## 1. Project Overview

OWASP Juice Shop (v19.2.1) is an intentionally vulnerable web application used for security training, CTF competitions, penetration testing practice, and awareness demonstrations. It encompasses vulnerabilities from the entire OWASP Top Ten and many other real-world security flaws, making it a deliberate security anti-pattern showcase. The application is an open-source OWASP Flagship project maintained by volunteers with a worldwide contributor base.

**Business context:** not available (no docs/business-context.md)
**Compliance scope:** Intentionally non-compliant — the application exists to demonstrate insecure patterns. OWASP Top Ten coverage is an explicit goal, not a compliance failure.

## 7. Security-Relevant Code

### 7.1 Auth & Session
**Mechanism:** JWT (RS256 via `jsonwebtoken`) stored as cookie (`token`); in-memory token map in `lib/insecurity.ts`; optional TOTP 2FA via `otplib`
**Key files:** `lib/insecurity.ts:56`, `routes/login.ts:23-26`, `routes/2fa.ts`
**Observations:**
- JWT signed with RS256 using a **hardcoded RSA private key** embedded as a string literal in `lib/insecurity.ts:23` — any attacker who reads the source can forge tokens
- Token stored in memory map (`authenticatedUsers.tokenMap`); server-side session revocation is possible but not tied to standard session middleware
- `express-jwt@0.1.3` (very old version, pinned) used for JWT validation middleware; `cookie-parser('kekse')` uses a hardcoded signing secret at `server.ts:289`

### 7.2 Authorization
**Mechanism:** Custom role-based middleware; JWT claims checked via `security.isAuthorized()`, `security.isAccounting()`, `security.appendUserId()` from `lib/insecurity.ts`
**Key files:** `routes/login.ts:74`, `routes/deluxe.ts`, `routes/orderHistory.ts`, `server.ts:622`
**Observations:**
- Role field (`role`) in JWT payload — if JWT is forged, attacker gains any role
- `/rest/admin/application-version` and `/rest/admin/application-configuration` have no auth middleware at `server.ts:604-605`
- `/rest/user/authentication-details` lists all active sessions/tokens — no authorization guard observed

### 7.3 Data Access
**Pattern:** Mix of raw Sequelize SQL strings (SQLi-vulnerable) and Sequelize model methods; MarsDB NoSQL for reviews
**Key files:** `routes/login.ts:34`, `routes/search.ts:23`, `routes/recycles.ts`
**Observations:**
- **SQL injection** in `routes/login.ts:34`: `SELECT * FROM Users WHERE email = '${req.body.email || ''}'` — direct string interpolation, no parameterization
- **SQL injection** in `routes/search.ts:23`: `SELECT * FROM Products WHERE ((name LIKE '%${criteria}%'` — user-controlled `q` param injected directly
- Sequelize model methods (`findOne`, `findAll`, `findByPk`) used safely in other routes; contrast with the raw query anti-patterns above

### 7.4 Input Handling
**Key files:** `routes/chatbot.ts`, `routes/b2bOrder.ts`, `routes/profileImageUrlUpload.ts`, `routes/fileUpload.ts`
**Observations:**
- `req.body` and `req.query` used without sanitization in multiple routes before SQL construction
- `profileImageUrlUpload.ts:24` makes a server-side `fetch(url)` with caller-controlled URL — **SSRF** vulnerability
- File upload route (`fileUpload.ts`) referenced in server — extension and MIME type checks exist but have bypass challenges

### 7.5 Serialization
**Key files:** `routes/fileUpload.ts`, `routes/chatbot.ts`, `routes/verify.ts`
**Observations:**
- `js-yaml` used to parse YAML uploads (`handleYamlUpload` in `server.ts:127`) — version 3.14 is the safe branch that disables unsafe `!!` tags by default
- `JSON.parse` used in `routes/chatbot.ts` and `routes/languages.ts` on data loaded from files; not on direct user input in most cases
- `libxmljs2` used for XML parsing (XML upload challenge) — known to have XXE-capable configurations

### 7.6 Crypto & Secrets
**Key files:** `lib/insecurity.ts:23,43-44,56`
**Observations:**
- **Hardcoded RSA private key** (2048-bit) on line 23 of `lib/insecurity.ts` — used to sign all JWT tokens
- **MD5 used for password hashing** (`crypto.createHash('md5')`) at `lib/insecurity.ts:43` — MD5 is not a password hashing function, trivially crackable
- HMAC-SHA256 with **hardcoded key** `'pa4qacea4VK9t9nGv7yZtwmj'` at `lib/insecurity.ts:44`
- `cookie-parser` initialized with hardcoded secret `'kekse'` at `server.ts:289`

### 7.7 Error Handling
**Key files:** `routes/verify.ts`, `server.ts`, `routes/login.ts`
**Observations:**
- Express `errorhandler` middleware used in development (`require('errorhandler')` at `server.ts:133`) — may expose stack traces
- `next(error)` pattern used throughout routes — error details forwarded to Express error handler without sanitization
- `console.error` used in startup timing wrapper (`server.ts:155`) — could log sensitive startup errors

### 7.8 Dangerous Sinks
**Key files:** `routes/b2bOrder.ts:23`, `routes/userProfile.ts:62`, `frontend/src/app/track-result/track-result.component.html`
**Observations:**
- **RCE via eval**: `routes/b2bOrder.ts:23` uses `vm.runInContext('safeEval(orderLinesData)', sandbox)` — `notevil` sandbox is bypassable; `orderLinesData` comes from `req.body`
- **SSTI/eval**: `routes/userProfile.ts:62` calls `eval(code)` where `code` is derived from the user's username matching pattern `#{(.*)}` — Server-Side Template Injection
- **XSS via innerHTML**: Angular components use `DomSanitizer` bypass patterns for intentional XSS challenges; `frontend/src/app/track-result/track-result.component.html` renders raw HTML

### 7.9 OAuth / OIDC
