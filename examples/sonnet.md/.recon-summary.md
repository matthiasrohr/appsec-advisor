## 1. Project Overview

OWASP Juice Shop v19.2.1 is an intentionally vulnerable Node.js/Express web application used as a security training, awareness, and CTF platform. It deliberately implements OWASP Top 10 and many other vulnerability classes as interactive challenges. It is NOT a production application template — all vulnerabilities are by design.

**Business context:** Open-source OWASP flagship project for security training. Deployed publicly at demo.owasp-juice.shop.
**Compliance scope:** Not specified (training/demo context; GDPR data erasure feature present).

## 7. Security-Relevant Code

### 7.1 Auth & Session
**Mechanism:** RSA-signed JWT (RS256, jsonwebtoken + express-jwt 0.1.3), Google OAuth2, 2FA/TOTP optional
**Key files:** lib/insecurity.ts:22-57, routes/login.ts, routes/2fa.ts
**Observations:**
- JWT private key hardcoded in source (`lib/insecurity.ts:23`) — entire RSA key embedded in code
- Login route (`routes/login.ts:34`) uses raw SQL template literal (SQL injection by design)
- `express-jwt` version 0.1.3 is extremely outdated (current is 8.x); algorithm validation not enforced
- Cookie parser uses hardcoded secret `'kekse'` (`server.ts:289`)
- No rate limiting on `/rest/user/login` endpoint

### 7.2 Authorization
**Mechanism:** JWT role checks via custom middleware (isAuthorized, isAccounting, isDeluxe, denyAll)
**Key files:** lib/insecurity.ts:54-55, server.ts:355-445, routes/verify.ts
**Observations:**
- Role stored in JWT payload; no server-side session/role binding (role tampering via forged JWT risk)
- `/api/Products/:id` PUT is deliberately unauthenticated (changeProduct challenge, server.ts:369)
- Some routes use `appendUserId()` middleware but no IDOR check — user can access other users' resources
- IP allowlist for accounting at `server.ts:430` uses fake IP `123.456.789`

### 7.3 Data Access
**Pattern:** Sequelize ORM (parameterized) for most models + raw SQL for login/search
**Key files:** routes/login.ts:34, routes/search.ts (inferred), lib/insecurity.ts
**Observations:**
- Login query uses string interpolation: `SELECT * FROM Users WHERE email = '${req.body.email}'` — SQLi
- ORM used for most CRUD operations (safe by default)
- MongoDB used for product reviews (mongoosejs); NoSQL injection patterns possible

### 7.4 Input Handling
**Key files:** routes/fileUpload.ts, routes/b2bOrder.ts, routes/userProfile.ts
**Observations:**
- File type validation by extension only (no MIME/magic-byte check), `routes/fileUpload.ts:67-73`
- XML upload enables XXE via `libxml.parseXml(data, { noent: true })`, `routes/fileUpload.ts:83`
- YAML upload uses `yaml.load()` (js-yaml, potentially unsafe), `routes/fileUpload.ts:117`
- ZIP upload vulnerable to path traversal (ZipSlip), `routes/fileUpload.ts:43-45`
- No CSRF protection (CORS is fully open wildcard)

### 7.5 Serialization
**Key files:** routes/fileUpload.ts:83 (XML), routes/fileUpload.ts:117 (YAML), routes/b2bOrder.ts
**Observations:**
- XML parsed with external entity resolution enabled (`noent: true`) — XXE by design
- B2B endpoint uses `vm.runInContext(safeEval(orderLinesData))` — Node.js VM sandbox escape risk
- YAML upload parsed with `js-yaml v3.14` `yaml.load()` which executes JS in YAML (unsafe schema)

### 7.6 Crypto & Secrets
**Key files:** lib/insecurity.ts:22-44
**Observations:**
- MD5 used for password hashing (`crypto.createHash('md5')`, `lib/insecurity.ts:43`) — critically weak
- HMAC uses hardcoded key `'pa4q****'` (`lib/insecurity.ts:44`)
- RSA private key hardcoded in source code (`lib/insecurity.ts:23`) — Critical
- `sanitize-html` version 1.4.2 is extremely outdated (current 2.x series)
- JWT signed with hardcoded private key that is in source control

### 7.7 Error Handling
**Key files:** server.ts, routes/fileUpload.ts:87
**Observations:**
- XML parse errors returned to client with xmlString content: `next(new Error('...deprecated...: ' + utils.trunc(xmlString, 400)))` — information disclosure
- Error details revealed in B2B order errors
- `errorhandler` npm package used (development-mode full stack trace exposure)
- Log files accessible at `/support/logs/` (directory listing)
