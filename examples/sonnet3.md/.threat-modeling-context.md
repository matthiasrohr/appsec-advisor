## Business Context

OWASP Juice Shop is an intentionally vulnerable web application used for security training, awareness, and demonstration. It deliberately implements nearly every major OWASP vulnerability category. The application functions as an online shop with user accounts, products, orders, baskets, wallets, payment methods, and an integrated chatbot. Despite being intentionally vulnerable, real-world deployments exist (demo.owasp-juice.shop) and may be exposed to real attacks.

## Architecture Notes

- Monolithic Node.js/Express backend with Angular SPA frontend
- SQLite database via Sequelize ORM  
- MongoDB-like MarsDB for product reviews and orders (in-memory)
- JWT-based authentication (RS256) using hardcoded RSA keys in source code
- CORS fully open (wildcard) - explicitly designed as a challenge
- No CSP headers configured via Helmet (xssFilter commented out)
- No HSTS headers
- Helmet used partially (noSniff, frameguard only)
- FTP directory with sensitive files exposed publicly
- Prometheus metrics endpoint exposed unauthenticated at /metrics
- Cookie parser uses hardcoded weak secret 'kekse'
- Passwords hashed with MD5 (intentional vulnerability)
- Private RSA key hardcoded in lib/insecurity.ts
- HMAC key hardcoded in lib/insecurity.ts
- No CSRF protection
- express-jwt version 0.1.3 (very old)
- jsonwebtoken version 0.4.0 (very old)
- sanitize-html version 1.4.2 (very old)
- unzipper version 0.9.15 (path traversal vulnerability)
- Angular auth tokens stored in localStorage (XSS risk)
- SBOM generation via CycloneDX in Docker build
- CodeQL SAST in CI (but with mutable @v3 tags for some actions)
- Rate limiting only on /rest/user/reset-password and /rest/2fa/* 
- No rate limiting on /rest/user/login (brute force risk)
- FTP files publicly browsable at /ftp
- /support/logs accessible for log file disclosure
- /encryptionkeys directory browsable
- errorhandler() middleware leaks stack traces
- XXE vulnerability in XML file upload (noent: true)
- SQL injection in login route and product search
- NoSQL injection in product review update and order tracking
- RCE via vm.runInContext in B2B order route
- SSRF via profile image URL upload
- Open redirect at /redirect
- bypassSecurityTrustHtml used in multiple components (XSS risk)
- Admin panel accessible by role claim in JWT (client-controlled)
- No BFF pattern - tokens in localStorage directly
