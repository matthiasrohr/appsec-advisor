# Recon Summary — OWASP Juice Shop

## 1. Business Context

OWASP Juice Shop v19.2.1 is an intentionally vulnerable web application used for security training, CTF competitions, and developer awareness. It is a full-stack JavaScript/TypeScript e-commerce application (Angular SPA + Node.js/Express backend) with deliberately introduced security vulnerabilities covering OWASP Top 10 and more. Licensed MIT, maintained by the OWASP community. This assessment documents all vulnerabilities as real security findings since the architecture decisions directly reflect real-world anti-patterns.

- **Primary Users**: Shoppers (customers), Administrators
- **Key Functionality**: User registration/login, product browsing/search, basket/checkout, reviews, feedback, data export (GDPR), admin panel, chatbot, crypto wallet (Web3), CTF challenges
- **Technology Stack**: Node.js 24 + TypeScript + Express 4.x + Angular 17+ SPA + SQLite (Sequelize) + NeDB/MongoDB-like (nedb-promises for reviews/orders)
- **Deployment**: Docker container (distroless base, non-root UID 65532), single container, port 3000

## 2. System Architecture

- **Complexity Tier**: Moderate (SPA frontend + REST API backend + dual database)
- **Components**:
  1. Angular 17+ SPA (frontend/src/) — client-side, served as static files
  2. Express REST API (routes/ + server.ts) — monolith, 61 route files, ~170 registered routes
  3. SQLite via Sequelize (models/) — primary database for users, products, orders
  4. NeDB (data/mongodb.ts) — secondary NoSQL store for reviews and orders
  5. CI/CD Pipeline (.github/workflows/) — GitHub Actions with 10+ workflows

## 7. Security Findings

### 7.1 IAM / Authentication

- **[CRITICAL]** Hardcoded RSA private key in lib/insecurity.ts:24 — embedded as string literal in source code. JWT tokens signed with this key; key is public in GitHub repository.
- **[HIGH]** No rate limiting on POST /rest/user/login — brute force attacks unrestricted.
- **[HIGH]** SQL injection in login query: `lib/insecurity.ts:34` — `SELECT * FROM Users WHERE email = '${req.body.email}'` — direct string interpolation.
- **[MEDIUM]** MD5 used for password hashing: `lib/insecurity.ts:44` — `crypto.createHash('md5')` — cryptographically broken.
- **[MEDIUM]** Change password endpoint (GET /rest/user/change-password) accepts current password as optional — can change password without knowing current password (routes/changePassword.ts:42).
- 2FA (TOTP) support implemented: routes/2fa.ts.
- OAuth2 login flow available: /oauth2/authorization endpoint.

### 7.2 Authorization

- **[HIGH]** IDOR on basket access — GET /rest/basket/:id does not verify basket belongs to requesting user. Only tracks challenge completion (routes/basket.ts:22).
- **[HIGH]** PUT /api/Products/:id authentication commented out: `server.ts:369` — `// app.put('/api/Products/:id', security.isAuthorized())` — unauthenticated product modification.
- **[MEDIUM]** Mass assignment vulnerability — POST /api/Users allows setting role field (server.ts:509) — allows privilege escalation to admin.
- Role-based access control (RBAC) present: roles.customer, roles.admin, roles.accounting, roles.deluxe.
- isAdmin(), isAccounting(), isDeluxe() middleware implemented.

### 7.3 Data Protection

- **[CRITICAL]** MD5 password hashing — no salt, collision-prone, rainbow table vulnerable (models/user.ts:70).
- **[HIGH]** Passwords stored as MD5 hashes without salting.
- **[MEDIUM]** JWT tokens stored in cookies AND localStorage (frontend) — XSS-accessible.
- **[LOW]** Sensitive data in URL query parameters — changePassword endpoint passes current/new passwords in GET query string (routes/changePassword.ts:13).

### 7.4 Secret Management

- **[CRITICAL]** RSA private key hardcoded in lib/insecurity.ts:24 — 1024-bit key embedded as string literal.
- **[HIGH]** HMAC key hardcoded: `lib/insecurity.ts:46` — `'pa4qacea4VK9t9nGv7yZtwmj'`.
- **[HIGH]** Cookie secret hardcoded: `server.ts:289` — `cookieParser('kekse')`.
- **[MEDIUM]** ctf.key file in repository root — CTF scoring key.
- **[MEDIUM]** encryptionkeys/ directory contains jwt.pub and premium.key served via /encryptionkeys/:file endpoint.

### 7.5 Frontend Security

- **[HIGH]** bypassSecurityTrustHtml used in multiple Angular components:
  - frontend/src/app/about/about.component.ts:119 (feedback display)
  - frontend/src/app/last-login-ip/last-login-ip.component.ts:39 (lastLoginIp from JWT payload)
  - frontend/src/app/administration/administration.component.ts:60, 78
  - frontend/src/app/search-result/search-result.component.ts:132, 170 (search query and product descriptions)
- **[MEDIUM]** Wildcard CORS: `server.ts:181-182` — `app.use(cors())` — no origin restriction.
- **[LOW]** Content-Security-Policy only applied to /profile page, not globally.

### 7.6 Output Encoding

- **[HIGH]** SQL queries use direct string interpolation — no parameterized queries in search.ts and login.ts.
- **[HIGH]** Multiple bypassSecurityTrustHtml calls bypass Angular's built-in XSS protection.
- Partial: sanitize-html used in some places, but sanitizeLegacy() is intentionally weak (lib/insecurity.ts).
- username field uses sanitizeLegacy() when challenge is enabled — intentionally vulnerable.

### 7.7 Audit & Logging

- **[MEDIUM]** Winston logger configured for Console transport only — no persistent log storage configuration at application level.
- **[MEDIUM]** Access logs via Morgan ('combined' format) — written to file with rotation, but no SIEM integration.
- **[LOW]** No security event logging for authentication failures, authorization violations.
- Prometheus metrics exposed at /metrics endpoint (unauthenticated by default).

### 7.8 Infrastructure & Network

- **[HIGH]** SSRF vulnerability — profileImageUrlUpload accepts arbitrary URLs and fetches them server-side (routes/profileImageUrlUpload.ts:27).
- **[MEDIUM]** Wildcard CORS with no origin allowlist.
- **[MEDIUM]** X-Powered-By disabled (`app.disable('x-powered-by')`), X-Frame-Options set (SAMEORIGIN), noSniff() active.
- **[LOW]** No HSTS header configured.
- **[LOW]** Feature-policy header set, but CSP not globally applied.
- Prometheus /metrics endpoint accessible without authentication.

### 7.9 Dependency & Supply Chain

- **[MEDIUM]** package-lock.json not found in repository — dependency integrity not guaranteed.
- **[LOW]** Most GitHub Actions SHA-pinned; codeql-analysis.yml has 3 actions pinned to @v3 mutable tags.
- **[LOW]** npm postinstall script runs frontend build and TypeScript compilation — supply chain risk if compromised.
- SBOM generation via cyclonedx-npm integrated in Docker build.
- Distroless base image (gcr.io/distroless/nodejs24-debian13) — good image hygiene.
- No npm audit or Snyk in CI pipeline detected.

### 7.10 Security Testing

- OWASP ZAP DAST scan configured (.github/workflows/zap_scan.yml).
- CodeQL static analysis configured (.github/workflows/codeql-analysis.yml).
- Mocha server-side tests (test/server/**/*.ts).
- Cypress E2E tests (test/cypress/).
- No npm audit blocking CI on Critical/High vulnerabilities.

### 7.11 OAuth/OIDC Implementation

- OAuth2 login endpoint present: server.ts references oauth2 flow.
- /oauth2/authorization endpoint available.
- JWT signed with RS256 (RSA-SHA256) using hardcoded key pair.
- No PKCE enforcement detected.
- No state parameter validation confirmed.

### 7.13 LLM / AI Patterns

### 7.14 CVE Scanning

- No npm audit or CVE blocking step in CI pipeline.
- CodeQL static analysis present but not dependency vulnerability scanning.

### 7.15 Lockfile Pinning

- package-lock.json not confirmed present (not found in root scan).
- CI uses `npm install` not `npm ci` — no lockfile integrity enforcement.

### 7.16 CI/CD Action Pinning

- MOSTLY SHA-pinned: ci.yml uses commit SHA for all actions.
- codeql-analysis.yml uses @v3 mutable tags for github/codeql-action/* (3 actions unpinned).
- coverallsapp/github-action@v2 in ci.yml — mutable tag.

### 7.17 Container Image Hygiene

- Final image: gcr.io/distroless/nodejs24-debian13 — good hygiene, minimal attack surface.
- NOT pinned to digest — version tag only.
- Non-root user (UID 65532) enforced.
- Multi-stage build with installer/runtime separation.
