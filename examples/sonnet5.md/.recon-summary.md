# Recon Summary — OWASP Juice Shop

## Section 1: Business Context

OWASP Juice Shop is a deliberately insecure web application for security training, CTF competitions, and tool demonstrations. It simulates a full e-commerce platform with user accounts, products, orders, payments, wallets, feedback, and a B2B API. Version: 19.2.1. Despite being intentionally vulnerable, the architectural patterns represent real-world anti-patterns critical for a production security assessment.

## Section 2: Technology Stack

- **Runtime:** Node.js 20-24 (LTS)
- **Backend Framework:** Express.js ^4.22.1 / TypeScript
- **Frontend:** Angular ^20.1.0 SPA (TypeScript)
- **Primary Database:** SQLite via Sequelize ORM ^6.37.3
- **Secondary Database:** MarsDB (MongoDB-like, in-process) for orders/reviews
- **Authentication:** express-jwt 0.1.3 (ancient) + jsonwebtoken 0.4.0 (ancient)
- **Template Engine:** Pug (server-side profile rendering)
- **File Upload:** multer
- **Security Headers:** helmet (partial), cors (wildcard), feature-policy
- **Metrics:** prom-client (Prometheus)
- **Logging:** winston (console transport only)
- **Container:** Docker (distroless/nodejs24-debian13)
- **SBOM:** CycloneDX generated during Docker build

## Section 4: Architecture Summary

- **Tier:** Moderate (monolith with clear frontend/backend separation + two database systems)
- **Pattern:** Single Express.js application serving both Angular SPA and REST API
- **No BFF:** SPA communicates directly with the API, JWT stored in localStorage
- **Single process:** API logic, file serving, template rendering all in one Node.js process
- **No API gateway / WAF in front**
- **External dependencies:** download module (chatbot training), fetch() for SSRF (profile image URL upload)

## Section 7: Security Findings

### 7.1 Authentication (IAM)
- `express-jwt` version 0.1.3 (extremely outdated, known vulnerabilities)
- `jsonwebtoken` version 0.4.0 (extremely outdated, algorithm confusion possible)
- RSA keys: public key read from file (encryptionkeys/jwt.pub), private key hardcoded in source
- Custom JWT auth, no SSO/OIDC, no standard identity provider
- In-memory token storage (authenticatedUsers.tokenMap) — no persistence, no revocation

### 7.2 Authorization
- `security.isAuthorized()` middleware on most authenticated routes
- `security.appendUserId()` injects UserId from JWT — but wallet/memory routes trust req.body.UserId
- Admin endpoints at `/rest/admin/*` — no admin role check on application-version, application-configuration
- `GET /api/Users` exposed to any authenticated user (no admin check)

### 7.3 Data Protection
- Passwords hashed with MD5 (`crypto.createHash('md5')`) — critically weak
- Security answers hashed with HMAC-SHA256 using hardcoded secret `pa4qacea4VK9t9nGv7yZtwmj`
- No encryption at rest for SQLite database
- User email in orders obfuscated with vowel replacement (not encrypted)

### 7.4 Secret Management
- RSA private key hardcoded in `lib/insecurity.ts:23` (1024-bit, begins with `-----BEGIN RSA PRIVATE KEY-----`)
- HMAC secret hardcoded in `lib/insecurity.ts:44` (`pa4qacea4VK9t9nGv7yZtwmj`)
- Cookie secret hardcoded in `server.ts:289` (`kekse`)
- CTF key in `ctf.key` file in repo root (30 bytes)
- No secret manager integration

### 7.5 Frontend Security
- JWT stored in `localStorage` (request.interceptor.ts:13, navbar.component.ts:233)
- `bypassSecurityTrustHtml()` used in 8+ Angular components (systemic XSS)
- CORS: `app.use(cors())` without origin restriction = wildcard, credentials not restricted
- No CSP headers (server-level; per-route CSP only on profile page, includes `unsafe-eval`)
- XSS filter commented out: `// app.use(helmet.xssFilter())`
- Angular 20 with strict mode but sanitizer bypassed manually

### 7.6 Secret Leakage
- Private key in source code (lib/insecurity.ts:23)
- HMAC secret in source (lib/insecurity.ts:44)
- Cookie secret in source (server.ts:289)
- CTF key file in repo

### 7.7 Input Validation
- SQL injection in login.ts:35 (raw string concatenation: `SELECT * FROM Users WHERE email = '${req.body.email}'`)
- SQL injection in search.ts:22 (raw string concatenation into SELECT query)
- XXE in fileUpload.ts (libxml with `noent: true`)
- RCE via eval() in userProfile.ts:55 (username passed directly to eval)
- RCE via vm.runInContext + notevil in b2bOrder.ts (authenticated)
- SSRF in profileImageUrlUpload.ts (unfiltered URL fetch)
- Zip Slip in fileUpload.ts handleZipFileUpload (path traversal in zip extraction)
- YAML bomb in fileUpload.ts (DoS via malicious YAML)
- Open redirect in redirect.ts (bypassable allowlist)
- No input size limits on most endpoints
- No schema validation layer

### 7.8 Output Encoding / XSS
- bypassSecurityTrustHtml in about.component.ts:119 (feedback comments)
- bypassSecurityTrustHtml in administration.component.ts:60,78 (user emails + feedback)
- bypassSecurityTrustHtml in search-result.component.ts:132,170 (product description + search query)
- bypassSecurityTrustHtml in last-login-ip.component.ts:39 (last login IP)
- bypassSecurityTrustHtml in track-result.component.ts:48 (order ID)
- bypassSecurityTrustHtml in score-board.component.ts:86 (challenge description)
- bypassSecurityTrustHtml in data-export.component.ts:57 (captcha image)
- Server-side: sanitizeLegacy() uses weak regex in models/user.ts for username

### 7.9 OAuth / OIDC / SSO
- None present. Custom JWT-based auth only.

### 7.10 SPA / BFF Architecture
- No BFF pattern. Angular SPA holds JWT in localStorage.
- Token passed as Authorization: Bearer header on every request
- No HttpOnly cookie for auth token
- No CSRF protection (cookie-parser with secret 'kekse' but no CSRF middleware)

### 7.11 Error Handling
- stack traces potentially exposed via Express error handler
- API errors return detailed messages (SQL errors, file paths)
- errorhandler package installed (development middleware that exposes full stack traces)

## Section 8: Additional Findings

### Sensitive File Exposure
- `/ftp` directory serves arbitrary files unauthenticated (intentional challenge)
- `/encryptionkeys` directory serves JWT public key unauthenticated
- `/support/logs` serves application access logs unauthenticated
- `ctf.key` in repository root with mode 664

### Password Issues
- MD5 password hashing (completely broken)
- Security questions with guessable answers
- Password change via GET request (CSRF-prone)

### SSRF
- profileImageUrlUpload.ts: `fetch(url)` with user-controlled URL — SSRF vector

### Zip Slip
- fileUpload.ts: Zip extraction with partial path traversal protection (checks absolutePath.includes(resolve('.')))

### SSTI / RCE
- userProfile.ts:55: `eval(code)` where code extracted from username matching `#{(.*)}` pattern
- b2bOrder.ts: `vm.runInContext('safeEval(orderLinesData)', sandbox)` — sandbox escape possible
